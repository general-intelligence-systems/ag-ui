# 04 — A2UI (Tier-3 generative UI)

A2UI lets the agent compose UI from a catalog of components. Our frontend renders
it (see the host-app `ai/canvas/` + the panel's `useRenderActivityMessage` path).
The server side is a **middleware** that must be ported.

References:
- `docs/concepts/generative-ui-specs.mdx` (the A2UI operation spec)
- `sdks/python/a2ui_toolkit/ag_ui_a2ui_toolkit/{validate,recovery}.py` (server helpers)
- CopilotKit's `@ag-ui/a2ui-middleware` (`node_modules`) — the exact transform we replace
- host-app: `sidecars/copilotkit/server.mjs` (`a2ui: { injectA2UITool, schema, defaultCatalogId }`)

## What the middleware does (port this)

1. **Inject a `render_a2ui` tool** into the LLM's tool list, whose parameters are
   driven by the **catalog schema**. The catalog is our own vocabulary, authored
   in `app/javascript/ai/catalog/definitions.ts`, emitted to JSON, and served at
   `GET /api/copilotkit/catalog`. The Ruby server fetches it at boot exactly like
   `server.mjs` does today (retry loop; degrade if unreachable). Pin the surface
   id with `defaultCatalogId` = the catalog's `catalogId`
   (`host://ai-catalog`) so the client catalog resolves.
   - **`injectA2UITool` is mandatory** — without the tool the agent falls back to
     a state snapshot that renders nothing (learned the hard way; see the
     host-app `ai/canvas/NOTES-low-level-a2ui.md`).
2. **When the model calls `render_a2ui`**, do NOT treat it as a normal client
   tool. Instead convert the call's `a2ui_operations` payload into
   **`ACTIVITY_SNAPSHOT`** events:
   ```
   s.activity_snapshot(
     message_id: surface_message_id,
     activity_type: "a2ui-surface",
     content: { a2ui_operations: [ … ] },
     replace: true)   # flat event shape — confirmed against events.py + @ag-ui/core zod
   ```
   (The `role:"activity"` wrapper exists only on the **history** form, the
   `ActivityMessage`: `{ id, role:"activity", activityType, content }`.)
   The client's activity renderer (our `a2uiCanvasRenderer`, resolved by
   `useRenderActivityMessage`) picks these up, stashes the surface, and renders
   it in `<ArtifactCanvas>`.
3. Re-emitting: the middleware dedups on `surfaceId`. Emit `createSurface` once
   per surface; use `updateComponents` / `updateDataModel` for changes (see the
   client-side `SurfaceMessageProcessor` in the host-app canvas).

## `/info` must advertise A2UI

The client feature-detects A2UI from `GET /info`. Match the capabilities object
(`docs/concepts/capabilities.mdx`, `sdks/python/ag_ui/core/capabilities.py`) so
the provider knows the agent can emit `a2ui-surface` activities. Get this wrong
and the surfaces arrive but the client never wires the renderer.

## Message-stream truth

A2UI surfaces live in history as `role: "activity"` messages
(`activityType: "a2ui-surface"`, `content.a2ui_operations`). Our client reads
them straight from `agent.messages` (the canvas does this). So make sure
activity messages **persist in the thread history** you return on the next run,
not just as transient events.

## Order of implementation

A2UI is phase 4 (after text + tools). It's a self-contained middleware:
catalog-fetch → tool injection → tool-call→activity conversion. The client side
already works (we built it); the server just has to emit the right activity
events.
